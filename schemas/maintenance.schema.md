# Test Maintenance Report Schema

Maintenance reports are generated by the `test-maintainer` skill when checking for discrepancies between specs, test artifacts, and the live application.

## File Naming

`<feature-id>.maintenance.md`

**Location:** `.argus/reports/maintenance/`

## Frontmatter

```yaml
---
id: <feature-id>-maintenance
feature: <feature-id>
url: <live app URL or "offline">
timestamp: <ISO 8601>
spec_discrepancies: <number>
broken_selectors: <number>
stale_artifacts: <number>
new_in_app: <number>
missing_from_app: <number>
status: <healthy | needs-attention | critical>
---
```

## Sections

### 1. Summary
Overview: feature checked, live URL (if provided), counts of each finding type, overall health status.

Health status:
- **healthy** — No discrepancies, no broken selectors, no stale artifacts
- **needs-attention** — Minor discrepancies or stale artifacts found
- **critical** — Broken selectors or spec vs app mismatches found

### 2. Spec vs App Discrepancies
Table with columns:
| Element | Spec Says | App Shows | Test Status | Action Needed |
Action Needed: "CR or Bug?" — tester decides.

Only present when a live app URL is provided.

### 3. Broken Selectors
Table with columns:
| Page Object | Selector | Status | Suggestion |
Status: ✅ Found | ❌ Not found
Suggestion: Alternative selector found in DOM, if any.

Only present when a live app URL is provided.

### 4. Missing from App
Bullet list of elements specified in the feature file but not found in the live DOM.

### 5. New in App
Bullet list of elements found in the live app but not documented in any spec or CR. These may be undocumented features or recent additions.

### 6. Stale Artifacts
Table with columns:
| Artifact | Last Updated | Upstream Changed | Status |
Status: ⚠️ May be outdated | ✅ Current

Checks the dependency chain:
- feature.md → testcase.md → page objects → flows → suites

### 7. Recommendation
AI-generated summary with:
- Count of discrepancies needing resolution
- Count of undocumented elements
- Count of stale artifacts
- Suggested next actions

## Core Principle

> Argus trusts the spec, not the code. If the app doesn't match the spec, that's a finding (potential bug), not a reason to update tests.

## Tester Decision Points

After each finding, the tester decides:
- A) Spec is correct → file a bug against the developer
- B) Code is correct, spec is outdated → update the spec first, then regenerate tests
- C) There's a CR for this → feed the CR and update everything

## Offline Mode

When no URL is provided, the report includes only:
- Stale artifact detection (file timestamp comparison)
- Spec consistency checks (feature file references valid scenarios, test cases reference valid features)
- Sections 2–5 are marked "Requires live app URL"
