# Bug Report Schema

Bug reports are generated by the `bug-reporter` skill when tests fail or verification finds issues.

## File naming

`BUG-<NUMBER>.report.md` — sequential numbering per project.

## Structure

```
# BUG-[NUMBER]: [Title]

**Severity:** Critical | Major | Minor | Trivial
**Feature:** [feature-id]
**TestcaseID:** [TC-ID that failed, if applicable]
**Environment:** [browser, device, URL]
**Found by:** [skill name — test-runner | ui-verifier | accessibility-checker | manual]

## Description
Brief description of the issue.

## Steps to Reproduce
1. Numbered steps
2. That anyone can follow
3. To reproduce the bug

## Expected Result
What should happen according to the spec.

## Actual Result
What actually happens.

## Evidence
- Screenshot: [path or inline]
- Console errors: [if captured]
- Network response: [if relevant]

## Additional Context
- Related test cases: [TC-IDs]
- Related features: [feature-ids]
- Stack trace: [if available]
```

## Severity Guidelines

| Severity | Criteria |
|----------|----------|
| **Critical** | App crashes, data loss, security vulnerability, complete feature broken |
| **Major** | Feature partially broken, workaround exists but painful |
| **Minor** | Cosmetic issue, minor UX problem, edge case failure |
| **Trivial** | Typo, alignment off by 1px, non-user-facing issue |

## Notes
- Bug reports can be pushed to Jira via `jira-connector` or saved as markdown
- Each bug report should be self-contained — a developer can reproduce without additional context
- Always reference the spec requirement that is violated
