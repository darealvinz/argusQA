# Report Schema

Reports are generated by the `report-generator` skill. Four report types exist, each with a different scope and audience.

## Report Types

### Sprint Report
**File naming:** `sprint-<number>.report.md`
**When:** End of sprint
**Audience:** QA team, dev leads, product owners

**Sections:**
1. **Summary** — table with: features tested, total cases, passed, failed, blocked, bugs reported, critical bugs
2. **Coverage by Feature** — table per feature: cases, pass, fail, blocked, status icon
3. **Coverage by Testing Layer** — table per layer: cases, pass rate
4. **Coverage by Browser/Device** — table per browser/device: pass rate
5. **Bug Summary** — table of open bugs: ID, feature, severity, status
6. **UI Verification Status** — table per feature: design match percentage
7. **Accessibility Status** — table per feature: WCAG level, issues count
8. **Risks & Blockers** — auto-generated from blocked tests and critical bugs
9. **Recommendation** — AI-generated go/no-go based on quality gate thresholds

### Feature Report
**File naming:** `<feature-id>.feature-report.md`
**When:** After testing a single feature
**Audience:** Developer, QA, product owner

**Sections:**
1. **Summary** — total cases, passed, failed
2. **Test Cases by Layer** — table per layer: cases, pass, fail, details
3. **Failed Test Case Details** — table with: TestcaseID, step, data, expected, actual
4. **UI Verification** — design match %, mismatches listed
5. **Accessibility** — WCAG status, violations listed
6. **Bugs Filed** — table: ID, severity, title

### Daily Report
**File naming:** `<YYYY-MM-DD>.daily-report.md`
**When:** End of day
**Audience:** QA team, project manager

**Sections:**
1. **Today's Activity** — features tested, cases executed, passed, failed, bugs reported
2. **Progress** — table per feature: status, remaining cases
3. **New Bugs Today** — table: ID, feature, severity, title
4. **Tomorrow's Plan** — bullet list of planned work

### Release Readiness Report
**File naming:** `<version>.release-report.md`
**When:** Before a release
**Audience:** QA lead, dev lead, product owner, stakeholders

**Sections:**
1. **Verdict** — GO | CONDITIONAL GO | NO GO
2. **Quality Gate Status** — table per gate: criteria, actual, status icon
3. **Failing Quality Gates** — details on each failing gate with context
4. **Risk Assessment** — table: risk, impact, mitigation
5. **Recommendation** — AI-generated narrative with evidence
6. **Sign-off** — table: role, name, status (pending)

## Quality Gates (for Release Readiness)

Read from `.argus/config.yaml` under `reporting.qualityGates`:

| Gate | Config Key | Default |
|------|-----------|---------|
| Test case pass rate | `passingRate` | 95% |
| Critical bugs open | `maxCriticalBugs` | 0 |
| Major bugs open | `maxMajorBugs` | 3 |
| UI verification match | `uiMatchThreshold` | 95% |
| Accessibility level | `accessibilityLevel` | AA |

## Output Formats

| Format | Extension | Use Case |
|--------|-----------|----------|
| Markdown | `.report.md` | Default, readable in repo and PRs |
| HTML | `.report.html` | Shareable with stakeholders |
| PDF | `.report.pdf` | Formal documentation, audit trails |
| JSON | `.report.json` | Dashboard integration (Grafana, Jira) |

Configured in `.argus/config.yaml` under `reporting.defaultFormat` and `reporting.formats`.
